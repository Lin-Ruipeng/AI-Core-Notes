### 第 2 课：数值积分与累积误差控制 —— 对抗浮点漂移的底层博弈
在第一课中，我们看到了减法如何“吃掉”有效数字。这节课，我们来看加法在极端条件下的隐秘杀手：**大数吃小数**。
#### 1. 理论拆解：从连续积分到离散求和
在纯数学中，对加速度 $a(t)$ 积分求速度 $v(t)$ 是轻而易举的：
$$ v(t) = v_0 + \int_{0}^{t} a(\tau) d\tau $$
在计算机中，我们将连续时间离散化为时间序列 $t_0, t_1, ..., t_n$（传感器采样）。最基础的数值积分方法是**矩形法（欧拉积分）**或**梯形法则**。以梯形法则为例，在区间 $[t_{i-1}, t_i]$ 内：
$$ v_i \approx v_{i-1} + \frac{a_{i-1} + a_i}{2} \Delta t $$
本质上，这是一个长循环内的累加过程：$v = v + \Delta v$。
#### 2. HPC 数值计算思路与浮点陷阱（核心）
在 C++ 中，浮点数（`float` 或 `double`）的有效位数是固定的（`double` 约 15-17 位十进制）。当你把一个很大的数和一个很小的数相加时，小数的低位会被直接**截断丢弃**，因为大数占用了所有的有效数字空间。
**传感器场景映射**：
假设你正在处理一辆高速行驶的汽车上的 IMU 数据，汽车初始速度 $v_0 = 100.0 \text{ m/s}$（大数），而由于某种极其微小的坡道或风阻，产生了 $a = 0.00001 \text{ m/s}^2$ 的加速度（小数）。采样率 100Hz，单次速度增量 $\Delta v = 0.0000001 \text{ m/s}$。
当你执行 `v += delta_v` 时，由于 $v$ 已经是 100.0，而 `delta_v` 在第 7 位小数之后，`double` 为了对齐指数，必须把 `delta_v` 的有效数字右移，导致它被完全舍入为零！**积分了半天，速度纹丝不动。**
**HPC 杀手锏：Kahan Summation（补偿求和）**
为了解决这个问题，顶尖的数值计算库（甚至某些编译器的 `-ffast-math` 背后）会使用 Kahan 求和算法。它的核心思想是：用一个额外的变量 `c`（补偿值），**把每次加法中被丢弃的低位误差偷偷记下来，并在下一次加法中加回去**。
伪代码逻辑：
```text
sum = 0.0
c = 0.0   // 补偿误差的累加器
for y in input_array:
    y = y - c          // 1. 把上次丢失的误差先补回来
    t = sum + y        // 2. 进行常规加法（这里可能再次丢失精度）
    c = (t - sum) - y  // 3. 反向计算：理论上 t - sum 应该等于 y，如果不等于，差值就是这次丢失的误差，存入 c
    sum = t            // 4. 更新总和
```
这套算法以仅仅增加几次基础算术运算的代价，将长序列累加的精度提升到了接近 $O(1)$ 的常数级误差，是 HPC 领域最经典的底层优化之一。
#### 3. 传感器场景的延展警告
在真实的 IMU 处理中，即便使用了 Kahan 求和，由于加速度计本身存在微小的零偏（Bias，比如 $0.001 \text{ m/s}^2$ 的常量误差），积分 100 秒后，计算出的速度就会产生 $0.1 \text{ m/s}$ 的漂移；再积分求位移，位移误差会随时间的**平方**发散。这就是为什么纯惯导不能长时间工作，必须引入卡尔曼滤波进行外部修正的原因。

**1. 采用 `std::accumulate` 产生误差是正常且预料之内的吗？**
**绝对正常。** 因为在 IEEE 754 浮点数标准中，**加法是不满足结合律的**。数学上 $(a + b) + c = a + (b + c)$，但在计算机里，由于每次加法后都要进行舍入对齐，这两者算出来的结果往往是不同的。`std::accumulate` 的底层逻辑被死死绑定为从左到右的顺序累加（`(init + elem1) + elem2)...`），只要你传入的数据存在量级差异，它就必定触发大数吃小数。它只保证“按你给的顺序机械地加”，不保证“数学上的最优逼近”。
**2. 数据精度的问题就是要我们 HPC 工程师自己负责吗？**
**是的，责无旁贷。** 编译器（即使开了 `-O3`）和标准库都是“瞎子”，它们看到的只有寄存器里的 0 和 1，根本不知道你算的是 IMU 的速度，还是银行的利息。编译器甚至可能因为你的代码里有多个相同操作数，自作聪明地进行“公共子表达式消除”或重排指令，从而打乱你的浮点计算顺序，导致结果发生微小变化。只有我们工程师清楚数据的物理上下文（比如这里大数是 10000，小数是 1e-5），才有资格去决定是用普通求和、Kahan 求和，还是 pairwise 求和。
