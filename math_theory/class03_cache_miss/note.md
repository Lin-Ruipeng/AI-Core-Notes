### 第 3 课：矩阵乘法的内存博弈 —— 缓存命中率决定生死
前两节课，我们搞定了标量（单个浮点数）计算中的精度陷阱。但在传感器融合（如卡尔曼滤波的姿态矩阵、协方差矩阵）中，我们的核心操作对象是矩阵。
很多刚接触 HPC 的人会有一个错觉：“我写了三层 `for` 循环实现了 $C = A \times B$，算法复杂度是 $O(N^3)$，这就够了。”
**在 HPC 领域，只看算法复杂度是幼稚的。现代 CPU 计算一个浮点乘法只需要 3~5 个时钟周期，但从主存（RAM）拿一次数据却需要 200~400 个时钟周期！**
#### 1. 理论拆解：内存墙与缓存行
CPU 不会直接去 RAM 读数据，而是按块读取到 L1/L2/L3 缓存中。这个“块”叫 **Cache Line（缓存行）**，在 x86 架构下通常是 64 字节（刚好装下 8 个 `double`）。
*   **缓存命中**：CPU 需要的数据在缓存里，瞬间拿到。
*   **缓存未命中**：数据不在缓存里，CPU 干等几百个周期去 RAM 搬数据。
#### 2. HPC 数值计算思路与内存陷阱（核心）
在 C/C++ 中，二维数组在内存中是**行主序**连续排列的。
假设有一个 $3 \times 3$ 矩阵 `A[3][3]`，它们在内存里的真实排列是：`A[0][0], A[0][1], A[0][2], A[1][0]...` 像一字排开的队伍。
现在我们要算 $C = A \times B$。数学公式是：$C_{i,j} = \sum_k A_{i,k} \times B_{k,j}$
**直觉写法（最致命的 HPC 反模式）**：
```cpp
for (int i = 0; i < N; ++i)
    for (int j = 0; j < N; ++j)
        for (int k = 0; k < N; ++k)
            C[i][j] += A[i][k] * B[k][j];
```
*分析内存访问*：
*   `A[i][k]`：随着 `k` 增加，是在同一行内移动，**完美命中缓存**。
*   `B[k][j]`：随着 `k` 增加，是在同一列内跳跃！这意味着每次循环都要跨越一整行的内存距离去取一个数。**每次都是缓存未命中！** 你的 CPU 99% 的时间在发呆等内存，计算单元闲置。
**HPC 正统写法（ijk -> ikj 循环交换）**：
```cpp
for (int i = 0; i < N; ++i)
    for (int k = 0; k < N; ++k) {     // 交换 j 和 k 的位置
        double temp = A[i][k];        // 提到外层，减少内存读取
        for (int j = 0; j < N; ++j)
            C[i][j] += temp * B[k][j];
}
```
*分析内存访问*：
*   `A[i][k]`：连续访问，命中。
*   `B[k][j]`：随着 `j` 增加，在 `B` 的第 `k` 行内连续移动，**完美命中缓存！**
仅仅是交换了两个循环的名字，在 $N=1000$ 时，性能可能会差出 **10 倍到 50 倍**！
#### 3. 传感器场景映射
在 IMU 的 AHRS（航姿参考系统）算法中，我们经常需要做 $3\times3$ 的旋转矩阵更新或 $4\times4$ 的齐次变换矩阵乘法。虽然矩阵很小（无法塞满缓存），但如果是 200Hz 甚至 1kHz 的采样率，这种微观上的内存优化能极大降低单次更新耗时，为后续的磁力计补偿、卡尔曼更新腾出宝贵的 CPU 时间片。另外，如果不理解这一点，以后你在调用 Eigen 库时，根本看不懂它为什么要提供 `.lazyEval()` 或者存储选项。
