### 第 4 课：分支预测与无分支编程 —— 实时性的守护神
前三节课，我们解决了计算精度（第一、二课）和内存带宽（第三课）的问题。现在，我们进入 HPC 的最后一个微观核心：**CPU 流水线与控制流**。
#### 1. 理论拆解：CPU 的“猜你喜欢”与流水线清空
现代 CPU 不是一条指令执行完再执行下一条的。它像工厂流水线一样，把指令分成“取指、译码、执行、访存、写回”等多个阶段，同一时刻有十几条指令在不同阶段并行工作（这叫**超标量流水线**）。
当你的代码遇到 `if / else` 时，CPU 必须知道走哪条分支，才能把后面的指令送入流水线。但此时 `if` 的条件往往还没计算完，怎么办？
**答案：猜。** CPU 内部有一个硬件叫**分支预测器**。
*   **猜对了**：流水线满载运行，性能极高。
*   **猜错了**：CPU 必须把流水线里错猜的指令全部作废清空，重新从正确的分支取指令。这个惩罚极其惨重，大约浪费 **15~20 个时钟周期**。
#### 2. HPC 数值计算思路与传感器陷阱（核心）
分支预测器最擅长处理**有规律**的数据（比如 `for` 循环的结束条件总是 predictable 的）。但它最害怕的是**完全随机、毫无规律的布尔值**。
**传感器场景映射**：
在处理 IMU 或激光雷达的原始数据流时，我们经常需要做异常值剔除或限幅。比如，判断加速度是否超过物理极限（如 $\pm 16g$）：
```cpp
if (acc > 16.0f) acc = 16.0f;
else if (acc < -16.0f) acc = -16.0f;
```
如果传感器处于剧烈震动环境，数据在阈值上下疯狂随机跳跃，分支预测器的准确率会暴跌到接近 50%。这意味着有一半的 `if` 会猜错！你的实时系统会因为这个“隐形杀手”突然卡顿，导致 IMU 丢帧。
**HPC 杀手锏：无分支编程**
工程师的职责是用数学运算代替逻辑判断，消除 `if/else`。
对于上面的限幅操作，C++ 标准库提供了 `<cmath>` 中的 `std::fmin` 和 `std::fmax`。在底层，它们会被编译器直接映射为 CPU 的硬件指令（如 x86 的 `MINSS` / `MAXSS`，或者 ARM 的 `FMINV`）。这些指令**没有分支**，无论数据多随机，耗时永远是恒定的 1 个周期！
#### 3. 编译器的狡猾（重要警告）
很多初学者不知道，现代编译器（如 GCC/Clang 开启 `-O3` 时）非常聪明。如果它看到你写了一个简单的 `if/else` 赋值，它可能会自动帮你优化成无分支的**条件传送指令（CMOV）**。
所以，为了在作业中真正“逼出”分支预测失败的代价，我们的代码逻辑必须稍微复杂一点点，让编译器“不敢”轻易使用 CMOV，只能乖乖生成跳转指令。

### 📝 课后实战作业
请使用 C++ 完成以下任务：
**题目：对抗随机数据流 —— 分支限幅 vs 无分支限幅**
1.  **数据准备**：生成一个包含 **10,000,000（一千万）** 个 `float` 元素的 `std::vector<float>`。数据必须是**随机且均匀分布**在 `[-30.0f, 30.0f]` 之间的。（提示：使用 `<random>` 的 `std::mt19937` 生成随机整数，然后除以一个常数映射到浮点范围，或者直接用 `std::uniform_real_distribution<float>`）。
2.  **实现两种处理方式**（将原数组拷贝两份，分别处理）：
    *   **方式 A（分支限幅）**：手写嵌套 `if / else if`，将超过 `16.0f` 的截断为 `16.0f`，低于 `-16.0f` 的截断为 `-16.0f`。为了防止编译器过度优化掉分支，在 `else` 分支里加上一个无用的但不影响结果的运算，比如 `acc = acc + 0.0f;`。
    *   **方式 B（无分支限幅）**：纯使用 `<cmath>` 的 `std::fmax` 和 `std::fmin` 嵌套实现同样的限幅逻辑。
3.  **性能测试**：使用 `<chrono>` 分别测量这两种方式处理完一千万个数据所需的毫秒数。
4.  **编译要求**：这次作业**必须**使用 `g++ main.cpp -O3 -march=native` 进行编译（因为只有在 `-O3` 下，CPU 流水线的威力才会完全释放，`fmin/fmax` 才会被映射为单指令）。如果不加 `-O3`，两者的差距可能不明显。
5.  **代码要求**：
    *   不要在循环内打印任何东西。
    *   结果数组不需要打印，只需打印耗时。
