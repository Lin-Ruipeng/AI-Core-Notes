### 第 5 课：多核并行与数据竞争的深渊 —— OpenMP 初探
前四节课，我们把单核 CPU 的算力压榨到了极致：解决了精度（ALU）、内存（Cache）、控制流（Branch Predictor）。但当今的处理器，哪怕是你手机里的芯片，都是 8 核甚至 16 核。如果我们只用单核跑循环，就等于白白浪费了 80% 的算力。
在 HPC 领域，想让循环跑在多个核上，最经典、最轻量级的方法不是手搓复杂的 `std::thread`，而是使用编译器指令：**OpenMP**。
#### 1. 理论拆解：Fork-Join 模型与数据竞争
OpenMP 的运行机制是 **Fork-Join（分叉-汇合）**：
主线程遇到指令时，会 Fork 出一群工作线程；大家干完活后，在末尾汇合，由主线程继续往下走。你只需要在 `for` 循环前面加一行神奇的咒语：`#pragma omp parallel for`，编译器就会自动把循环的迭代次数平均切分给各个 CPU 核心。
**但是，这里潜伏着实时系统最恐怖的杀手：数据竞争。**
假设我们要把一千万个数累加：`sum += array[i];`
如果加了 OpenMP 并行，线程 A 和线程 B 可能同时读取到 `sum = 100`，各自加了 50，然后同时写回 `sum = 150`。
正确的应该是 200，结果却丢了 50！更可怕的是，这种错误**每次运行的结果都不一样**，完全无法复现和调试。
#### 2. HPC 数值计算思路与传感器陷阱（核心）
**传感器场景映射**：
在 IMU 的预处理阶段，我们经常需要计算一个滑动窗口内（比如 1000 个采样点）的均方根（RMS，用于判断震动强度）或方差。公式是：$RMS = \sqrt{\frac{1}{N} \sum x_i^2}$。这个求和过程就是数据竞争的高发区。
**HPC 杀手锏：OpenMP Reduction（归约）**
怎么解决竞争？加锁吗？**绝对不行！** 在一千万次的循环里加锁，锁的开销会让并行变成负优化（比单核还慢无数倍）。
OpenMP 提供了一个优雅至极的解决方案：`reduction(+:sum)`。
它的底层逻辑是：编译器会自动为**每一个线程**创建一个独立的、局部的 `sum` 副本。线程们各自安全地累加自己的副本，等循环结束后，编译器再在底层把所有副本加起来。全程无锁，性能狂飙。

### 📝 课后实战作业
请使用 C++ 完成以下任务：
**题目：万剑归宗 —— OpenMP 归约求 RMS**
1.  **数据准备**：生成一个包含 **10,000,000（一千万）** 个 `double` 元素的 `std::vector<double>`，数值随机分布在 `0.0` 到 `100.0` 之间。
2.  **实现三种求和方式**（用来计算 RMS 公式里的 $\sum x_i^2$ 部分）：
    *   **方式 A（单核基准）**：普通的 `for` 循环累加 `sum += val * val`。
    *   **方式 B（错误并行）**：加上 `#pragma omp parallel for`，但**不加** `reduction`。让 `sum` 作为一个外部变量被多线程疯狂竞争写入。*(注意：这个值算出来必定是错的，我们就是要看看错得有多离谱)*。
    *   **方式 C（正确并行）**：加上 `#pragma omp parallel for reduction(+:sum)`。使用归约模式。
3.  **计算与打印**：
    *   分别用方式 A、B、C 算出来的 sum，除以 $N$，再开平方（`std::sqrt`），得到三个 RMS 值并打印出来。
    *   打印方式 A（单核）和方式 C（正确并行）的计算耗时（毫秒级）。方式 B 不需要测耗时。
4.  **编译要求（极度重要）**：
    *   OpenMP 不是 C++ 标准库，而是编译器扩展。编译时**必须**加上 `-fopenmp` 标志。
    *   完整编译命令：`g++ main.cpp -O3 -march=native -fopenmp`
    *   *(如果你是 Mac 用户，Clang 默认不带 OpenMP，可能需要 `brew install libomp`，然后编译命令改为 `g++ main.cpp -O3 -march=native -Xpreprocessor -fopenmp -lomp`，如果环境不支持请告诉我，我们换纯 C++ 方案)*
5.  **代码要求**：
    *   必须包含头文件 `<omp.h>`。
    *   可以在 `main` 开头打印一下 `omp_get_max_threads()`，看看你的 CPU 被识别出了多少个线程。
