### 第 8 课（对应大纲第 6 课）：高斯分布的 C++ 采样与传感器噪声建模
前两个模块，我们处理的都是“确定性”的数学。但从这一刻起，我们要面对真实世界的灵魂——**不确定性**。
卡尔曼滤波之所以伟大，就是因为它敢于承认“我的传感器是瞎子”。在 C++ 中如何真实地模拟这个“瞎子”，直接决定了你离线训练和测试滤波器是否有效。
#### 1. 理论拆解：如何用确定性的代码生成随机的混沌？
计算机本质上是图灵机，只能做确定性运算，根本不存在“真随机”。我们只能用**伪随机数生成器（PRNG）**来逼近。
**数学映射：Box-Muller 变换**
在早期的数值计算中，人们用“逆变换采样法”生成高斯分布：先算出高斯累积分布函数（CDF）的逆函数（需要计算复杂的误差函数 `erf` 的逆），然后把均匀分布丢进去。**这在 HPC 中是绝对的性能毒药。**
现代 C++ `<random>` 底层使用的是 **Box-Muller 变换**：
假设你有两个独立的均匀分布随机数 $U_1, U_2 \sim U(0, 1)$，你可以通过极其简单的三角函数和对数，直接变换出两个独立的标准正态分布随机数 $Z_0, Z_1$：
$$Z_0 = \sqrt{-2 \ln U_1} \cos(2\pi U_2)$$
$$Z_1 = \sqrt{-2 \ln U_1} \sin(2\pi U_2)$$
计算复杂度从复杂的特殊函数，降维打击成了基本的加减乘除和 `sqrt`/`log`。
#### 2. HPC 性能陷阱：`<random>` 的隐秘开销
既然 C++ 标准库帮我们封装好了 `std::normal_distribution`，我们是不是可以闭着眼睛用了？
**错。**
在传感器仿真中（例如模拟 IMU 以 1000Hz 运行 1000 秒，生成 100 万个噪声点），如果你在循环内部这样写：
```cpp
for (int i = 0; i < 1000000; ++i) {
    std::normal_distribution<double> d(0, 0.01); // 每次循环都在重新构造对象！
    noise[i] = d(gen);
}
```
这会产生极其严重的性能损耗。`std::normal_distribution` 是一个**有状态**的对象（它内部保留了 Box-Muller 变换算出的第二个值 $Z_1$，留给下一次调用直接用，省掉一次三角函数计算）。如果你每次循环都重新构造，不仅破坏了状态，还增加了构造函数的开销。
#### 3. 多核噩梦：随机数引擎绝对不能共享
我们在第 5 课学了 OpenMP。如果你试图把生成随机数的循环并行化：
```cpp
std::mt19937 gen(rd());
#pragma omp parallel for
for(int i=0; i<N; ++i) {
    noise[i] = dist(gen); // 灾难！
}
```
**多线程同时调用同一个 `gen`，会引发恐怖的数据竞争，生成的噪声不再是高斯分布，而是一堆毫无规律的垃圾，甚至会导致段错误。**
在 HPC 中，标准做法是为**每一个线程分配一个独立的随机数引擎**，并且用不同的种子。

### 📝 课后实战作业：蒙特卡洛噪声注入与多线程加速
请编写 C++ 代码，模拟一个陀螺仪在静止状态下的高斯白噪声输出。
**任务要求**：
1.  **参数设定**：模拟时长 1000 秒，采样率 1000Hz，共生成 $N = 1,000,000$ 个噪声点。噪声满足标准正态分布 $\mathcal{N}(0, 1)$（均值为 0，方差为 1）。
2.  **单线程基准**：
    *   在循环**外部**实例化 `std::mt19937` 和 `std::normal_distribution<double>`。
    *   用 `chrono` 测量生成这 100 万个点并存入 `std::vector<double>` 的耗时。
    *   *(附加)* 粗略计算一下这 100 万个数的均值和方差，验证是否符合 $\mathcal{N}(0,1)$。
3.  **多线程陷阱版（必踩坑）**：
    *   用 `#pragma omp parallel for` 并行化上面的循环，**只保留一个全局的 `gen`**。
    *   测量耗时（可能会变快，也可能崩），并打印计算出的均值和方差。**观察这时的方差是不是崩坏了。**
4.  **多线程正确版（HPC 真经）**：
    *   使用 OpenMP 的机制，确保每个线程都有自己的 `gen` 和 `dist` 实例。
    *   *提示*：你可以在 parallel 区域内声明局部变量，或者使用 C++11 的 `thread_local` 关键字，或者在 parallel for 的 `firstprivate` 子句中传入（但注意引擎不能拷贝，最好在并行块内部用 `omp_get_thread_num()` 作为种子局部创建）。
    *   测量正确并行化后的耗时，并再次验证均值和方差。
5.  **编译**：记得加上 `-fopenmp`。
**提交代码和三种情况（单线程、错误多线程、正确多线程）的耗时与方差结果。我在此等待，不会继续往下讲。**
