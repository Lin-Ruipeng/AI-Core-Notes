### 第 11 课（对应大纲模块四·第 8 课）：Welford 算法 —— 拯救在线计算的精度灾难
欢迎进入模块四：数理统计与滤波算法的最终实战收网。从这一刻起，我们面对的敌人不再是数学公式，而是**端侧设备的物理限制**。
#### 1. 端侧痛点：不能存，只能算
假设你在写无人机 IMU 的初始化代码。为了算出当前这个廉价陀螺仪的真实噪声，你需要让它静止 10 分钟，采样率 1000Hz，这意味着你要处理 600,000 个数据点。
在自动驾驶的底层板子上，内存极其宝贵。你**绝对不允许**把这 60 万个 `double` 存进一个 `std::vector` 里再去求均值和方差（那会占用好几 MB 内存，且容易引发缓存失效）。
你必须像流水线一样：**读一个数据，更新一下统计量，然后立刻把这个数据扔掉。** 这叫“在线递推计算”。
#### 2. 经典公式的“精度灾难”
一提到在线算方差，所有刚学过概率论的人都会写出这个“移位公式”：
$$Var(X) = E[X^2] - (E[X])^2 = \frac{1}{n}\sum x_i^2 - \left(\frac{1}{n}\sum x_i\right)^2$$
这个公式在数学考试里能拿满分，但在 C++ 工程里是**绝对的毒药**。
**灾难推演**：
假设 IMU 的真实加速度是 $1000000.0\ m/s^2$（比如处于剧烈加速的火箭上），而噪声标准差只有 $0.1$。
你用 `double` 去累加：
*   $E[X^2]$ 算出来大概是 $1.0 \times 10^{12}$
*   $(E[X])^2$ 算出来也大概是 $1.0 \times 10^{12}$
在 C++ 中，`double` 只有 15~17 位有效十进制数字。当你做减法 $10^{12} - 10^{12}$ 时，前面 12 位全部抵消了（灾难性相消，Catastrophic Cancellation），剩下的有效数字全是浮点截断误差带来的垃圾！
结果是什么？**方差变成了负数！** （方差在物理上绝不可能为负，一旦为负，后续的卡尔曼滤波协方差矩阵直接崩溃，程序直接炸毁）。
#### 3. HPC 救世主：Welford 算法
1962 年，B. P. Welford 提出了一种只需要单遍扫描、且完全避免了大数相减的递推算法。它不存历史数据，也不算 $E[X^2]$，而是动态维护两个变量：当前均值 $M_k$ 和当前平方和 $S_k$。
当读入第 $k$ 个数据 $x_k$ 时（假设从 1 开始计数）：
1. 计算当前值与旧均值的差：$\delta = x_k - M_{k-1}$
2. 更新均值：$M_k = M_{k-1} + \frac{\delta}{k}$
3. 更新平方和（极其精妙的一步）：$S_k = S_{k-1} + \delta \times (x_k - M_k)$
4. 最终方差：$Var = \frac{S_k}{k - 1}$ （除以 $k-1$ 是为了无偏估计）
**为什么它不会炸？**
看第 3 步，$\delta$ 是当前数据与均值的差，它巧妙地剥离了那个巨大的“基础值（比如 1000000）”，$\delta$ 本身非常小（比如就在 -0.3 到 +0.3 之间）。你累加的永远是极小数字的乘积，完美规避了 `double` 的精度上限！

### 📝 课后实战作业：见证方差变负数的灵异事件
请用 C++ 写一个程序，**不使用任何 vector 存储数据**，模拟流式读取。
**任务要求**：
1.  **模拟极端传感器**：设置真实均值为 `1e9`（十亿），真实标准差为 `1.0`。
2.  **生成流式数据**：用一个简单的正态分布随机数发生器，循环 $N=100000$ 次。每次生成一个数据，直接喂给计算逻辑，**不要存进数组**。
3.  **方法 A（经典毒药公式）**：
    维护两个变量 `sum_x` 和 `sum_x_sq`。每次读入数据就累加它们。
    循环结束后，计算 `mean = sum_x / N`，计算 `var = (sum_x_sq / N) - (mean * mean)`。
4.  **方法 B（Welford 神功）**：
    维护 `M` (初始 0.0) 和 `S` (初始 0.0)。
    按照上面讲的 3 步递推更新。
    循环结束后，计算 `var_welford = S / (N - 1)`。
5.  **报告**：
    打印方法 A 算出的方差。
    打印方法 B 算出的方差。
    *(提示：为了防止编译器过度优化把中间步骤吃掉，请使用 `-O3` 编译，并且注意观察方法 A 的输出是不是一个极其荒谬的数字甚至负数)*
